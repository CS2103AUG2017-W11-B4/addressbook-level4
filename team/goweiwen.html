Enhancement Added: Alias

External behavior

Start of Extract [from: User Guide]

Adding aliases : alias

If you find yourself forgetting your commands often, you can define aliases to rename the commands to something easier to remember!

Unify’s alias command creates an alias to another command. If an alias is unspecified, it lists all aliases.

Format: alias [ALIAS COMMAND]

  • alias unfriend delete 1
    unfriend (deletes the first person in the list)

  • alias friends find t/friend
    friends (lists all persons with the friend tag)

  • alias
    Lists all your previously defined aliases.


Deleting aliases : unalias

Deletes a previously defined alias.

Format: unalias ALIAS

  • alias unfriend delete
    unalias unfriend
    unfriend
    The unfriend command fails as there is no longer such a command.

End of Extract

Justification

Aliases help users execute commands they might forget. For example, if the user can’t remember how to add a new contact to the address book, they can type any of add, create, or new. In addition, we allow users to define their own aliases if the default ones are not memorable enough.

We do not provide short-form aliases (e.g. a as an alias for add) by default as we feel having a powerful autocompletion feature completely eclipses any uses for cryptic short-form commands. If the user feels short-form aliases are required, they can add them themselves.

Implementation

Start of Extract [from: Developer Guide]

Aliases

We implement aliases by adding a new Aliases class to the UserPrefs model. Since UserPrefs is saved to file, aliases will persist through application restarts.

Aliases contains a HashMap<String, String> which maps alias → command. This allows us to find the matching command in amortized O(1) time. By default, Aliases contains a map of synonyms for frequently used commands. For example, new → add, remove → delete, show → list, etc.

We added a new parser, parseCommandAndArguments in ParserUtil that parses user input into a commandWord and arguments, which AddressBookParser uses. To allow access to the UserPrefs instance in ParserUtil, we changed UserPrefs into a Singleton. parseCommandAndArguments refers to the Aliases instance inside UserPrefs and returns the mapped command.

Design Considerations

Aspect: Accessing UserPrefs from ParserUtil and other classes
Alternative 1 (current choice): Turn UserPrefs into a Singleton
Pros: Easy to implement. Minimise code change across code base.
Cons: Increases coupling. Difficult to stub in testing. Does not reset between tests.
Alternative 2: Pass the UserPrefs instance down to the parser
Pros: Less coupling. Follows Law of Demeter.
Cons: Heavy changes to code base to pass the instance downstream.

Enhancement Added: Command Box Validation

External behavior

Start of Extract [from: User Guide]

  • Don’t fret! Unify offers inline hints to prompt you what to type next,

    and will validate your input and display icons matching your command as you type.

End of Extract

Justification

In order to familiarise new users to the command-line interface, we validate commands as the users type. This allows them to know if their commands are incorrect before they press enter. This is only helpful for users able to touch-type. As our target audience is computing students, we can safely assume a majority of the users will benefit from this feature.

Implementation

Start of Extract [from: Developer Guide]

Command Box Validation

We validate the user’s command as he is typing, which allows real-time feedback to the user on whether their command is valid.

This is done by listening to the TextField associated to CommandBox. Whenever there is a change in input of the TextField, we call setStyleByValidityOfInput. As we have a reference to the LogicMangager (which is used for undo/redo), we can just run LogicManager#parse to check if the current input is valid, and style the input box accordingly.

Design Considerations

Aspect: Listen to CommandBox input’s changes
Alternative 1 (current choice): Listen directly to commandTextField.
Pros: No coupling with EventsCenter.
Cons: Inconsistent with implementation of other features (e.g. hints, command box icon).
Alternative 2: Subscribe to CommandInputChangedEvent.
Pros: Easily extensible as logic can be extracted into another class.
Cons: Adds additional dependency to EventsCenter.

End of Extract

Enhancement Added: Command Box Icon

External behavior

Start of Extract [from: User Guide]

  • Don’t fret! Unify offers inline hints to prompt you what to type next,

    and will validate your input and display icons matching your command as you type.

End of Extract

Justification

As with input validation, this feature is introduced to give users a hint of what their command will do, before they execute it. Besides helping users use the command-line more effectively, it also helps to make the user interface more polished, as these icons are present in most search boxes (e.g. internet browsers display the page’s icon, macOS’s Spotlight displays the application’s icon).

Implementation

Start of Extract [from: Developer Guide]

Command Box Icon

In addition to styling the command box, we also show the user icons matching their command on the right of the command box.

We leverage a third-party library called Ikonli. We add a new JavaFX component, CommandBoxIcon, which subscribes to CommandInputChangedEvent and calls parseCommandAndArguments, and passes the returned commandWord into parseIconCode, which then returns an Ikon, which we display.

Design Considerations

Aspect: Where to get icons
Alternative 1 (current choice): Use Ikonli, a third-party library.
Pros: Easy to implement.
Cons: Adds additional third-party dependency.
Alternative 2: Embed them as images or icons.
Pros: Easy to customize, as icons are all embedded images.
Cons: Reinvents the wheel.

End of Extract

Implementation

Start of Extract [from: Developer Guide]

Autocomplete

We implement autocompletion by introducing a new class: Autocomplete. This class contains all the utility methods required for obtaining an appropriate autocompletion from a given user input. The main method is autocomplete(String input). This takes in the user’s input and attempts to generate a hint from it. As we need to display the autocompletion in the command box hints, this reduces duplicated code required for producing the autocompletion.

There are several other utility methods in the class, such as autocompleteFromList, which takes in an input and an array of strings, and returns the first command that contains input as a prefix, or autocompleteCommand, which takes in a command word and compares it with a list of all commands and aliases, and autocompletes the command.

Enhancement Proposed: History Substring Search

Currently, when pressing Up or Down in the command box, we can cycle through our previously typed commands, irregardless of what is typed in the command box. One possible enhancement is changing it so it only cycles through commands that contain whatever is in the command box as a substring (i.e. like zsh-history-substring-search).

Enhancement Proposed: Composite Aliases

The alias feature currently implemented only allows users to create aliases for one partial or full command. One possible enhancement is to allow users to delimit commands in their aliases with semicolons, like: alias delete3 delete 1; delete 1; delete 1. This will allow users to perform repetitive tasks quickly, with just one command.
